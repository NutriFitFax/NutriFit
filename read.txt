NutriFit next steps
===================

Current status
--------------
- Backend is deployed on Render:
  https://nutrifit-backend-lnm0.onrender.com
- Backend endpoints currently available:
  GET /health
  GET /barcode/{barcode}
  GET /search?q=&page=&page_size=
  POST /estimate-meal
  GET /meal-plan?time_frame=&target_calories=&diet=
  GET /recipes/{id}
- Barcode lookup uses USDA first, then OpenFoodFacts fallback.
- Search uses USDA.
- Meal photo uses OpenAI. It must not return fake foods when OpenAI fails.
- Meal plan and recipe details use Spoonacular.
- Flutter API client exists in mobile/lib/api/.
- Camera, barcode, meal photo, search, food detail, home, navigation, and history UI exist.
- Flutter is not installed on this machine, so mobile tests must be run by someone with Flutter installed.
- No Supabase/database connection exists in the backend yet. There are no JDBC,
  Postgres, Supabase, repository, or migration files in this repo.

New feature: Spoonacular meal planner
-------------------------------------
Backend and Flutter API client are now wired. Flutter UI screens are still a team task.

1. Spoonacular API key is created and should be stored in Render.
   - Go to https://spoonacular.com/food-api/docs
   - Create an account and copy the API key.
   - Store it only in backend environment variables, for example:
     SPOONACULAR_API_KEY=...
   - Do not put the key in Flutter or git.

2. Backend config is implemented.
   - SPOONACULAR_API_KEY and SPOONACULAR_BASE_URL are in AppSettings.
   - They are documented in .env.example and render.yaml.

3. Backend models are implemented.
   - MealPlanResponse
   - PlannedMeal
   - RecipeDetails
   - RecipeNutrients

4. Backend service is implemented.
   - SpoonacularClient

   Useful endpoints:
   - GET https://api.spoonacular.com/mealplanner/generate
     Params:
     timeFrame=day or week
     targetCalories=<number>
     diet=<optional>
   - GET https://api.spoonacular.com/recipes/{id}/information
     Params:
     includeNutrition=true

5. Backend controller endpoints are implemented.
   - GET /meal-plan?time_frame=day&target_calories=2000&diet=vegetarian
   - GET /recipes/{id}

6. Backend tests are implemented.
   - Use MockWebServer.
   - Test successful meal plan.
   - Test Spoonacular failure -> 502.
   - Test missing key -> clear error.

7. Flutter API methods are implemented.
   - api.generateMealPlan(...)
   - api.getRecipeDetails(...)
   - Add Dart models matching backend JSON.

8. Add Flutter UI.
   Suggested folder:
   - mobile/lib/features/meal_plan/
   Screens:
   - MealPlanScreen
   - RecipeDetailScreen
   Required states:
   - loading
   - success
   - empty
   - API quota/error


New feature: exercise calorie burning with METs
-----------------------------------------------
This does not need an external API. Use a local lookup table from the Compendium of Physical Activities.

Sources:
- https://pacompendium.com/
- https://pacompendium.com/adult-compendium

1. Decide where activity logs live.
   Suggested local storage table/model:
   - id
   - activityName
   - met
   - durationMinutes
   - userWeightKg
   - caloriesBurned
   - loggedAt

2. Add a MET lookup table.
   Start small:
   - walking_slow: 2.8
   - walking_brisk: 4.3
   - running_easy: 7.0
   - cycling_light: 4.0
   - cycling_moderate: 6.8
   - strength_training: 3.5
   - yoga: 2.5
   - swimming_moderate: 5.8

3. Add calculation function.
   Formula:
   calories = (MET * 3.5 * weightKg / 200) * durationMinutes

4. Add Flutter service/model.
   Suggested files:
   - mobile/lib/features/tracking/activity_met.dart
   - mobile/lib/features/tracking/activity_log.dart
   - mobile/lib/features/tracking/calorie_burn_calculator.dart

5. Add UI.
   Suggested screen:
   - ActivityLogScreen
   Inputs:
   - activity dropdown
   - duration minutes
   - user weight from profile/latest weight
   Output:
   - estimated calories burned
   - save log button

6. Connect to home screen.
   - Show calories burned today.
   - Optionally subtract burned calories from net calories.

7. Add tests.
   Example:
   - 5 MET, 70 kg, 30 min = 183.75 kcal


Supabase database connection
----------------------------
Current status: not connected in code.

What is missing:
- A backend database dependency/config.
- Supabase Postgres connection env vars in Render.
- Tables or migrations for users, meals, water, weight, activity, and history.
- Repository/service code that reads and writes those tables.

Safe setup approach:
1. In Supabase, use the project database connection string, not the public anon key,
   for backend server-to-database access.
2. Store it only in Render environment variables. Do not put it in Flutter or git.
3. Add a backend database health endpoint before adding app data writes.
4. Add migrations/schema after Davud finalizes the storage models.
5. Only then connect Flutter screens to backend persistence endpoints.
